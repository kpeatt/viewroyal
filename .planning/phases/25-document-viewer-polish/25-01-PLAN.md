**1. Add custom marked renderer for tables:**
Import `Renderer` from `marked`. Create a `defaultRenderer = new Renderer()`. Call `marked.use()` with a custom `renderer.table()` method that wraps the default table output in `<div class="table-scroll-container overflow-x-auto -mx-2 px-2 my-4">`. Keep `gfm: true, breaks: false`.

Per research (25-RESEARCH.md Pattern 1), use this exact pattern:
```typescript
import { marked, Renderer } from "marked";

const defaultRenderer = new Renderer();
marked.use({
  gfm: true,
  breaks: false,
  renderer: {
    table(...args: Parameters<typeof defaultRenderer.table>) {
      const html = defaultRenderer.table.apply(this, args);
      return `<div class="table-scroll-container overflow-x-auto -mx-2 px-2 my-4">${html}</div>`;
    },
  },
});
```

<task type="auto">
  <name>Task 2: Add table scroll fade indicator CSS</name>
  <files>apps/web/app/app.css</files>
  <action>
Add CSS for the table scroll fade indicator at the END of `app.css` (after the existing `@keyframes` blocks). This provides a white gradient overlay on the right edge of tables that overflow, hinting to the user that more content exists.

Per locked decision: "shadow/fade hints on the overflow edge". Per research recommendation: pseudo-element approach with white gradient overlay.

Add this CSS:
```css
/* Table scroll shadow indicators */
.table-scroll-container {
  position: relative;
}
.table-scroll-container::after {
  content: '';
  position: absolute;
  top: 0;
  right: 0;
  bottom: 0;
  width: 2rem;
  background: linear-gradient(to right, transparent, white);
  pointer-events: none;
  opacity: 0;
  transition: opacity 0.2s;
}
.table-scroll-container.has-overflow::after {
  opacity: 1;
}
```

The `has-overflow` class will be toggled by a client-side ResizeObserver in document-viewer.tsx (handled by Plan 25-02). Without JS, the fade is hidden by default -- which is correct since if JS is unavailable, the table still scrolls, it just lacks the visual hint.
