**Per user decision:** RAG agent tools receive municipality_id and scope their vector search + keyword search. Tool-level scoping is sufficient (no prompt-level scoping needed).

**rag.server.ts (1665 lines) -- Key architectural change:**

The RAG tools are currently defined as module-level closures. Per the research pitfall analysis, refactor to pass municipality_id through to every tool function.

1. Modify `runQuestionAgent` signature to accept `municipality: Municipality` (or at minimum `municipalityId: string`)
2. Pass municipalityId into the tool creation scope so every tool closure has access
3. For EVERY tool function that calls `.rpc("match_*", {...})` or `.rpc("hybrid_search_*", {...})`, add `filter_municipality_id: municipalityId` to the RPC params
4. For EVERY tool function that uses `.from()` queries directly (e.g., fetching meeting details, person data), add `.eq("municipality_id", municipalityId)` or appropriate `!inner` join
5. Specific tools to scope:
   - `search_motions` / `search_council_records` -- add filter_municipality_id to match_motions RPC
   - `search_matters` -- add filter_municipality_id to match_matters RPC
   - `search_agenda_items` -- add filter_municipality_id to match_agenda_items RPC
   - `search_key_statements` -- add filter_municipality_id to match_key_statements RPC
   - `search_transcript_segments` -- scope transcript segment search by municipality
   - `search_document_sections` -- add filter_municipality_id to match/hybrid RPC
   - `search_bylaws` -- add filter_municipality_id to match_bylaws RPC
   - `get_voting_history` -- scope votes via meeting municipality_id
   - `get_statements_by_person` -- scope key_statements by municipality
   - `getPerson` / person lookup -- scope via organization municipality

6. Any enrichment queries inside tools (e.g., fetching meeting title for a motion's meeting_id) should also be scoped, though they may already be inherently scoped if looking up by specific ID.

**api.ask.tsx:**
- Already imports `getMunicipality`. Obtain the municipality object in the action handler.
- Pass municipality to `runQuestionAgent(supabase, query, municipality, ...)` or however the refactored signature looks.

**api.search.tsx (if it also calls RAG):**
- Ensure municipality is passed to any RAG calls. Check if api.search.tsx calls `runQuestionAgent` -- if so, pass municipality there too.

IMPORTANT: The RAG file is 1665 lines. Read the FULL file before making changes. Understand the tool structure, the module-level getSupabase() pattern, and how tools are registered. Make surgical changes -- don't rewrite the entire file.
