Create `apps/docs/content/docs/guides/authentication.mdx` with frontmatter title "Authentication" and description "API key usage, headers, rate limits, and error handling".

Content structure (all details from RESEARCH.md -- verified against source code):

- **Overview paragraph**: All data endpoints require an API key. Read-only API, no write operations. Health endpoint is the only unauthenticated endpoint.

- **Getting an API Key section**: Link to https://viewroyal.ai/settings/api-keys. Describe key format: `vr_` prefix followed by 64 hex characters. Use `<Callout type="warn">` to warn that the key is shown only once at creation time (stored as SHA-256 hash). Mention the key management page allows creating multiple keys and revoking compromised ones.

- **Using Your API Key section**: Two methods:
  1. **Header (recommended)**: `X-API-Key: vr_your_key_here` -- show curl/JS/Python examples with Tabs groupId="language"
  2. **Query Parameter (fallback)**: `?apikey=vr_your_key_here` -- mention this is for situations where headers can't be set, note it's less secure (key visible in logs/URLs).
  Use `<Callout type="warn">` for security: "Never expose your API key in client-side JavaScript. Always make API calls from your backend server."

- **Rate Limiting section**: 100 requests per 60 seconds, scoped per API key (not per IP). Response headers: `X-RateLimit-Limit: 100` on every authenticated response. When limit exceeded: HTTP 429 with `Retry-After: 60` header. Show the 429 error response shape:
  ```json
  { "error": { "code": "RATE_LIMIT_EXCEEDED", "message": "Rate limit exceeded. Try again later.", "status": 429 } }
  ```
  Show a retry logic example in JS and Python using Tabs -- simple exponential backoff respecting Retry-After header.

- **Authentication Errors section**: Table of auth-specific error codes:
  | Status | Code | When |
  |--------|------|------|
  | 401 | `MISSING_API_KEY` | No X-API-Key header or apikey query param |
  | 401 | `INVALID_API_KEY` | Key not found or has been revoked |
  | 429 | `RATE_LIMIT_EXCEEDED` | Over 100 requests in 60 seconds |

  Show the 401 error response shape. Link to the full [Error Handling guide](/docs/guides/error-handling) for all error codes.

- **CORS section**: Brief note that the API supports CORS with `Origin: *`. Allowed methods: GET, HEAD, OPTIONS. Exposed headers: X-Request-Id, X-RateLimit-Limit, Retry-After. This means browser-based apps can call the API directly (but should proxy through a backend to protect the API key).

- **Best Practices section**: Bulleted list:
  - Store API key in environment variables, never in source code
  - Use header auth over query parameter
  - Implement retry logic with exponential backoff for 429 responses
  - Create separate API keys for separate applications
  - Revoke compromised keys immediately at /settings/api-keys

Use `import { Tab, Tabs } from 'fumadocs-ui/components/tabs';` and `import { Callout } from 'fumadocs-ui/components/callout';` at the top of the file.
